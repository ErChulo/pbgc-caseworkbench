PBGC Metadata Scraper Prompt v1.0
Version Date: 2026-04-29
Companion Use: Run Before Any R5 Scrape, Plan Summary, Data Element Listing, Benefit Programming, or Case Memo Work Product.

PURPOSE
Create a single authoritative metadata file for a trusteeshiped defined benefit pension plan. This is the first case setup step. The output stores global case variables and source-document attributes in one place so all downstream work products use the same plan name, dates, sponsor names, document chronology, covered groups, plan-year rules, freeze dates, and citation-backed metadata.

PRIMARY TASK
Read every file in the Source section end-to-end and produce one JSON file named metadata-test.json unless the user specifies another filename.

The scraper must identify:
1) case-level metadata for the trusteeshiped plan;
2) canonical normalized metadata that should be reused globally;
3) document-level metadata for every source file;
4) aliases and variants appearing across documents;
5) unresolved or conflicting facts requiring review;
6) exact citations for every extracted fact.

This metadata scraper is not an R5 benefit-provision scraper. Its purpose is to establish global plan attributes and a source-document registry for downstream PBGC work.

CORE PRINCIPLE
Do not leave an obvious global variable blank merely because no metadata.json was provided. If the plan name, sponsor name, original effective date, plan year, or document title appears in the plan documents, extract it, normalize it, and cite it.

AUTHORITY HIERARCHY
Use the following source authority order when facts conflict:

1) PBGC case-level sources, if available:
   - metadata.json;
   - CMS extract;
   - Trusteeship Agreement;
   - Trusteeship Working Group Memo;
   - Plan Document List;
   - PBGC Image Viewer index;
   - Notice of Determination, Notice of Intent to Terminate, or related trusteeship records.

2) Executed plan documents and signed/adopted amendments.

3) Board resolutions, adoption pages, signature pages, certifications, or trustee acknowledgments attached to plan documents.

4) Summary of Material Modifications, ERISA 204(h) notices, Summary Plan Descriptions, collective bargaining agreements, actuarial valuation reports, administrative policies, and correspondence.

5) File names and folder names.
Use file names only as hints for year, type, or source identification. Do not use file names as sole support for legal facts if the document text contradicts or does not support them.

If a lower-authority source conflicts with a higher-authority source, keep both values in conflicts[] and use the higher-authority value as canonical unless the higher-authority source is clearly obsolete or not applicable.

OUTPUT FILE
Save exactly one JSON file:
metadata-test.json

GENERAL FIELD OBJECT FORMAT
Every factual field that can be extracted from sources must use this object structure:

{
  "value": "...",
  "status": "extracted",
  "confidence": "high",
  "sources": [
    {
      "source_filename": "...",
      "document_id": "...",
      "section": "...",
      "page_pdf": 1,
      "page_printed": "...",
      "quote": "..."
    }
  ],
  "notes": ""
}

Allowed status values:
- extracted: directly found in a source.
- normalized: derived by normalizing one or more extracted variants.
- inferred_from_source: logically inferred from explicit source text; use sparingly and explain in notes.
- conflicting: multiple plausible values exist.
- unknown: not found in the available source set.
- not_applicable: affirmatively not applicable.

Allowed confidence values:
- high: explicit source text, clear citation, no material conflict.
- medium: explicit source text but ambiguity, variant wording, or lower-authority source.
- low: weak support, file-name support only, partial OCR, illegible page, or uncertain interpretation.
- none: unknown or not applicable.

For status unknown:
- value must be "".
- confidence must be "none".
- sources must be [].
- notes must briefly say what source would normally supply the field.

For status not_applicable:
- value may be "" or a short explanation.
- cite the source if the non-applicability is explicitly stated.

CITATION RULES
Every extracted, normalized, inferred_from_source, conflicting, or not_applicable field must include citations unless the value is derived solely from aggregation of already cited subfields.

Each source citation must include:
- source_filename;
- document_id;
- section, only if explicitly labeled;
- page_pdf, 1-based PDF page number if applicable;
- page_printed, if shown on page;
- quote, the shortest controlling excerpt sufficient to verify the field.

Do not guess page numbers or section numbers.
For text files with page markers, use text page markers.
For text files without page markers, page_pdf may be "" but quote must be present.

PROPER CASE AND NORMALIZATION
Use clean title case for normalized names, preserving acronyms:
PBGC, ERISA, IRS, IRC, DOPT, DOTR, BPD, NOIT, NOD, EIN, PN, QJSA, QPSA, CBA, SMM, SPD, AFTAP, FICA.

Keep exact source variants in aliases.
Example:
canonical_plan_metadata.canonical_plan_name.value:
"The College Of Saint Rose Non-Contract Employees' Pension Plan"

source_aliases.plan_name_aliases:
[
  "The College Of Saint Rose Non-Contract Employees' Pension Plan",
  "College Of St. Rose Non-Contract Employees' Pension Plan",
  "The College Of Saint Rose Non-Contract Pension Plan"
]

DOCUMENT TYPE RULES
Full plan documents/restatements:
- Extract exact title page plan name.
- Extract statement-of-purpose effective dates.
- Extract plan year, anniversary date, sponsor, trustee/funding agent, and participant groups when available.
- Mark document_role as full_restatement or original_or_base_plan.

Narrow amendments:
- Extract amendment number, amended plan name, effective date, adoption/board approval date, execution/signature dates, trustee/funding agent, and provisions touched.
- Do not treat amendment recital history as current plan state unless explicitly stated.
- Mark document_role as narrow_amendment or amendment.

204(h) notices and SMMs:
- Extract benefit freeze date, participation freeze date, freeze description, distribution date, and stated plan name.
- Mark document_type as 204h_notice or summary_material_modification as appropriate.
- Mark document_role as notice_or_summary.
- If the notice says official plan documents govern, include that in notes.

Collective bargaining agreements:
- Extract union name, bargaining unit covered group, period covered, and pension article references.
- Do not treat CBA pension language as controlling plan text unless it directly defines eligibility or participation in the pension plan and no higher-authority source contradicts it.
- Mark document_role as cba_context.

VALIDATION RULES
Before finalizing:
1) Every source file reviewed must appear in document_registry.
2) canonical_plan_metadata.canonical_plan_name must be populated if any source contains a plan name.
3) case_metadata.plan_name_for_pbgc_work_products must be populated if canonical_plan_name is populated.
4) Do not leave plan name blank merely because no metadata.json exists.
5) Every extracted field must have at least one citation.
6) Unknown fields must not contain guessed values.
7) File-name-derived values must have confidence low unless confirmed by source text.
8) Conflicts must not be silently resolved; document the selected canonical value and reason.
9) Chronology must include all extracted effective, adoption, execution, freeze, notice, and CBA period dates.
10) JSON must parse as valid JSON.

RECOMMENDED CANONICALIZATION RULES
For plan name:
- Prefer the latest full plan document title if it is consistent with prior full plan documents.
- Preserve older spellings and variants in source_aliases.plan_name_aliases.
- Do not remove legally meaningful qualifiers such as “Non-Contract Employees.”

For sponsor:
- Prefer the sponsor/employer/company name in the latest full plan document.
- Preserve historical sponsor aliases.

For plan year:
- Prefer explicit “Plan Year” or “Anniversary Date” definitions in the plan document.
- Do not infer plan year solely from fiscal year unless the document defines them consistently.

For freeze dates:
- Prefer signed plan amendment or 204(h) notice that explicitly states the freeze date.
- If only a notice is available, mark confidence medium and note that official plan amendment should be obtained or checked.

For document dates:
- Separate effective date, adoption date, execution date, board approval date, and signature date.
- Do not collapse them into one date.

FAILURE MODES TO AVOID
- Do not leave canonical_plan_name blank when the plan name appears on title pages.
- Do not treat case_name and plan_name as identical unless a source supports it.
- Do not infer DOPT, DOTR, BPD, NOIT, or NOD from plan provisions.
- Do not ignore title pages, preambles, recitals, signature pages, certifications, or board-resolution pages.
- Do not overwrite historical trustee names with the latest trustee name; preserve chronology.
- Do not classify a 204(h) notice as a full plan amendment unless the source contains executed amendment language.
- Do not treat CBA covered groups as plan-covered groups unless the plan document or CBA pension article supports the connection.
- Do not silently resolve “St. Rose” versus “Saint Rose”; normalize but keep aliases.
- Do not rely on file names as legal metadata.

DELIVERABLE RESPONSE
After saving metadata-test.json, provide:
1) a link to metadata-test.json;
2) a short warning list if any important fields are unknown or conflict;
3) no full JSON dump unless requested.
